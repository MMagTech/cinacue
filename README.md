# Movie Channel

A small, self-hosted **television-style movie channel** for Unraid. An
administrator schedules movies from an existing Plex library into a rolling
7-day calendar; the public sees a live, read-only channel that always joins the
currently-scheduled movie at its live position — like flipping to a TV channel.

This repository implements the **complete v1** (all eight milestones): the
foundation, Plex integration, schedule editing, the live FFmpeg stream engine,
the public television-style page, admin security hardening, logging, and Unraid
deployment. It runs end-to-end (backend + compiled frontend + SQLite + Docker).
The one thing unverified from this environment is NVENC on real hardware — see
the note at the end. See *Status* below.

---

## What works today

- **FastAPI backend** with SQLite persistence (SQLModel).
- **React + TypeScript + Vite frontend**, compiled and served by the backend.
- **Configuration via environment variables** (`app/config.py`, `.env.example`).
- **`/health`** endpoint.
- **Public, read-only API** (`/api/public/*`) — status, now-playing, upcoming,
  channel-config. Never returns Plex tokens, server URLs, or filesystem paths.
- **Administrator login** with bcrypt password hash, signed HTTP-only session
  cookie (SameSite=Lax), CSRF protection on writes, and login rate limiting.
- **Encoding settings model** (channel-wide): maximum resolution, video
  bitrate, audio bitrate, H.264 NVENC encoder, encoder preset.
- **No-upscaling resolution calculation**, unit-tested against every example in
  the brief plus aspect-ratio, even-dimension, and portrait edge cases.
- **Dockerfile** (NVIDIA CUDA base, FFmpeg + FFprobe, Node build stage,
  non-root user, health check) and an Unraid-ready **docker-compose.yml**.
- **Plex integration**: connect, search the movie library, read full metadata,
  translate Plex paths to the mount, verify the source file, and proxy posters
  (the Plex token stays server-side).
- **Schedule editing**: add a movie from Plex (end time computed from runtime),
  edit its time, move it to another day, delete it — all with overlap
  validation (no auto-shifting) and CSRF-protected writes.
- **Live stream engine**: an FFmpeg process manager (states, log capture,
  retry, orphan cleanup, no-shell args) driven by a background controller that
  starts the scheduled movie at the correct live offset, transitions between
  movies, stops during gaps, and recovers from failures. HLS output is served
  at `/stream`. Admin Start/Stop, live dashboard, and a diagnostics endpoint
  (Plex, mount, FFmpeg, FFprobe, NVENC) are included.
- **Public television page**: live HLS player (no usable seek bar), now-playing
  poster/title/times/progress, upcoming list, and an OFF AIR state.
- **Security hardening**: HttpOnly + SameSite session cookie (Secure flag
  configurable for HTTPS), CSRF on writes, login rate limiting, baseline
  security headers, and public responses that never leak secrets or paths.
- **Logging** to stdout of connection/schedule/stream/auth/NVENC events, with
  secrets (token, password, cookies, app secret) deliberately never logged.
- **104 passing unit tests** (including a real software-FFmpeg HLS encode).

## Run locally (development)

You need Python 3.10+ and Node 18+.

**1. Backend**

```bash
pip install -r requirements.txt

# First run needs an admin password to seed the account.
export ADMIN_PASSWORD="choose-a-strong-password"
export CONFIG_DIR=./data/config
export STREAM_DIR=./data/stream

uvicorn app.main:app --reload --port 8000
```

The backend serves the compiled frontend from `app/static` if present. For a
pure backend run without a build, API and `/health` still work.

**2. Frontend**

```bash
cd frontend
npm install
npm run build      # outputs into ../app/static — the backend then serves it
```

Now open <http://localhost:8000/> for the public page and
<http://localhost:8000/admin> for the admin login.

For a hot-reloading frontend during development, run `npm run dev` (Vite proxies
`/api` and `/health` to the backend on port 8000) and open the Vite URL.

**Run the tests**

```bash
python3 -m pytest -q
```

## Run on Unraid (Docker)

**1. Build and start**

```bash
# From the project directory on the Unraid host:
docker compose build
ADMIN_PASSWORD="choose-a-strong-password" docker compose up -d
```

The compose file maps host port **8090** to the container's 8000.

**2. Volumes (edit paths to match your server)**

| Container path      | Host mapping                          | Purpose                                  |
| ------------------- | ------------------------------------- | ---------------------------------------- |
| `/config`           | `/mnt/user/appdata/movie-channel`     | SQLite DB, app secret, admin hash        |
| `/media/movies`     | `/mnt/user/media/movies` *(read-only)*| Your movie library                       |
| `/stream`           | `/tmp/movie-channel`                  | RAM-backed HLS segments                  |

`/tmp` on Unraid is tmpfs (RAM), which is exactly what you want for HLS output.

**3. NVIDIA GPU**

The compose file requests the GPU via both `runtime: nvidia` and the modern
`deploy.resources` reservation. You need the **Nvidia Driver** plugin on Unraid
and the NVIDIA Container Toolkit. `NVIDIA_VISIBLE_DEVICES=all` and
`NVIDIA_DRIVER_CAPABILITIES=compute,video,utility` are set for you.

**4. First login**

Set `ADMIN_PASSWORD` for the first start only. Once the hash is written to
`/config`, remove the variable and restart. There is **no default password**.

## The no-upscaling rule

The single most important rule: **the channel never enlarges a movie beyond its
source resolution.** The configured resolution is a *maximum*, not a target.
Aspect ratio is always preserved; output is never cropped or stretched; final
width and height are always even (required by H.264/yuv420p).

`app/encoding.py::calculate_output_dimensions` implements this and is exercised
by `tests/test_no_upscaling.py`, covering:

```text
3840x2160, max 1080p  -> 1920x1080
1920x1080, max 1080p  -> 1920x1080
1280x720,  max 1080p  -> 1280x720   (no enlargement)
720x480,   max 720p   -> 720x480    (no enlargement)
1920x800,  max 720p   -> 1280x532   (aspect preserved, no crop)
```

### A note on NVIDIA encoding

**NVENC has not been verified on real hardware from this environment.** The
Dockerfile installs FFmpeg with NVENC support and the entrypoint reports whether
`h264_nvenc` is listed, but *listing* an encoder is not the same as a working
GPU encode. Per the brief, the application will surface NVENC availability to
the administrator via the diagnostics endpoint (a later milestone) and will not
silently fall back to CPU encoding. Confirm NVENC works on your Unraid host with:

```bash
docker exec movie-channel ffmpeg -hide_banner -f lavfi -i testsrc=size=1280x720:rate=30 \
  -t 2 -c:v h264_nvenc -f null -
```

A successful run (no CUDA/NVENC errors) confirms hardware encoding.
